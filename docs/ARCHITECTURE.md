# URL Shortener — Architecture Documentation

## Overview

A URL shortening service built with FastAPI, PostgreSQL, and Redis. Users submit a long URL and receive a short code that redirects to the original. Designed with caching as a first-class concern given the read-heavy nature of URL shorteners.

---

## System Architecture

```
Client
  │
  ├── POST /urls/          → API Layer → PostgreSQL (write)
  │                                    → Redis (cache on write)
  │
  └── GET  /urls/{code}    → API Layer → Redis (cache check)
                                       → PostgreSQL (cache miss only)
                                       → 301 Redirect
```

---

## Components

### API Layer (FastAPI)
Handles all HTTP requests, input validation, and routing. Pydantic models enforce data integrity at the boundary — `HttpUrl` ensures no malformed URLs enter the system.

### Database (PostgreSQL)
Primary source of truth. Stores all URL mappings with auto-incrementing IDs used to generate short codes.

**Schema:**
```sql
CREATE TABLE urls (
  id         SERIAL PRIMARY KEY,
  short_url  VARCHAR UNIQUE,
  long_url   TEXT UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Cache (Redis)
Sits in front of the database on the read path. Given that URL shorteners have a read:write ratio of roughly 100:1, caching the redirect lookup has significant impact.

**Cache key:** `{short_code}`
**Cache value:** `{long_url}`
**TTL:** 3 minutes (180 seconds)

**Measured performance:**
| Path | Latency |
|------|---------|
| Cache miss (DB query) | ~3000ms |
| Cache hit (Redis) | ~17ms |

That is a **~176x speedup** on cached requests.

## ID Generation — Base62 Encoding

Short codes are generated by Base62 encoding the database's auto-increment ID.

**Character set:** `0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ`

**Why Base62?**
- URL-safe — no special characters
- Compact — 6 characters can represent ~56 billion unique URLs (62⁶)
- Collision-free — each ID is unique by definition

**Why not hashing?**
Hash functions can produce collisions — two different URLs mapping to the same short code. Using an auto-increment ID as the source eliminates this entirely.

**Example:**
```
ID 1       → "1"
ID 61      → "Z"
ID 62      → "10"
ID 9999999 → "FXsk"
```

## Key Design Decisions

### 1. Two-phase write on POST
Currently the POST creates a URL record, gets the auto-increment ID, encodes it to Base62, then updates the record with the short code. This is two database round trips per write.

**Known improvement:** Use PostgreSQL's `RETURNING id` clause to get the ID in the same INSERT query, then do a single UPDATE — or restructure to avoid the UPDATE entirely by computing the short code before the second commit.

### 2. Cache-on-write
When a URL is shortened, it is immediately written to Redis. This means the first redirect request after creation is a cache hit, not a DB query.

### 3. List endpoint cache invalidation
The `GET /urls/` pagination endpoint caches results by `limit:skip` key. This means newly added URLs will not appear in cached list responses until TTL expires.

**Known trade-off:** Stale list results for up to 3 minutes after a write. Acceptable for this use case. A production fix would be to invalidate or update list caches on every POST.

### 4. Duplicate URL handling
Before inserting, the POST endpoint checks if the long URL already exists and returns `409 Conflict` if so. This prevents duplicate entries for the same URL.

---

## What Would Break First at Scale

1. **Single PostgreSQL instance** — no replication, no read replicas. Under heavy write load this becomes the bottleneck.
2. **Redis TTL too short** — at 3 minutes, popular URLs will frequently expire and cause DB hits. TTL should be tuned based on access patterns — frequently accessed URLs should have longer TTLs.
3. **No connection pooling** — under concurrent load, DB connections will become exhausted. PgBouncer or SQLAlchemy's connection pool settings would need tuning.
4. **Sequential ID generation** — predictable short codes mean users can enumerate all URLs by incrementing the code. A production system would add randomness or authentication.
